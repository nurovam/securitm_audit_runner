# Manual Checks Policy

Часть требований в базовом профиле `met_rekom_linux` не автоматизирована полностью.
Для таких пунктов агент возвращает `SKIP` и явно сообщает, что требуется ручной аудит.

## Почему Это Допустимо

- Некоторые требования требуют контекстной оценки прав доступа и владельцев.
- Автоматизация таких проверок без дополнительных правил даёт много ложных срабатываний.
- Для production-использования ручные проверки должны быть явно отделены от автоматических.

## Текущие Manual Checks

- `met_2_3_2_running_process_file_perms`
- `met_2_3_3_cron_jobs_file_perms`
- `met_2_3_4_sudo_exec_file_perms`
- `met_2_3_8_system_bins_libs_perms`

## Кандидаты На Автоматизацию

Для этих проверок есть внешние инструменты, которые можно использовать как основу
для следующей итерации автоматизации:

- `met_2_3_2_running_process_file_perms`
  - основной кандидат: `osquery`
  - полезные источники: `processes`, `process_open_files`
  - цель: получать реальные бинарники и связанные пути запущенных процессов, а затем проверять владельца и права уже в Python-коде агента.

- `met_2_3_3_cron_jobs_file_perms`
  - основной кандидат: `osquery`
  - полезные источники: `crontab`
  - цель: извлекать реальные cron entry, разбирать вызываемые команды и проверять права на связанные файлы/скрипты.

- `met_2_3_4_sudo_exec_file_perms`
  - основной кандидат: `cvtsudoers` из проекта `sudo`
  - цель: нормализовать `sudoers` и include-файлы, выделить допустимые команды и проверить владельца/права на целевые исполняемые файлы.

- `met_2_3_8_system_bins_libs_perms`
  - основной кандидат: `AIDE`
  - цель: использовать baseline по критичным системным путям и атрибутам файлов вместо грубого единоразового сканирования.

Дополнительный инструмент:
- `auditd`
  - полезен там, где нужен не только снимок состояния, но и фиксация фактических исполнений через `cron`, `sudo` и `execve`.

Практический минимальный стек для будущей автоматизации:
1. `osquery`
2. `cvtsudoers`
3. `AIDE`

## Профиль По Умолчанию

`configs/audit.yml.example` по умолчанию **не** включает manual-only проверки.

Причина:
- они всегда возвращают `SKIP` и не повышают качество автоматического baseline;
- в open-source first-run они только увеличивают шум в отчёте;
- включать их лучше осознанно, когда оператор хочет видеть напоминания о ручной ревизии.

Если нужно добавить их в профиль, перечислите соответствующие `check_id` в
`audit.checks.enabled` вручную.

## Правило Для Развития Проекта

Для каждой проверки со статусом `SKIP` нужно выбрать один из трёх путей:

1. Автоматизировать проверку.
2. Оставить её manual и явно документировать это.
3. Исключить её из default-профиля, если автоматизация невозможна или слишком шумная.
